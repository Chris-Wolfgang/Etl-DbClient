Remarks

Two transaction modes are supported:

  • Caller-managed — pass a DbTransaction to the constructor. The loader uses it but never commits or rolls back. The caller is responsible for transaction lifetime.
  • Auto-managed — pass null for the transaction parameter. The loader creates its own transaction, commits on success, and rolls back on exception.

The caller owns the DbConnection lifetime — the loader does not open, close, or dispose it. The connection must be open before calling LoadAsync.

Thread safety. A DbLoader<TRecord> instance is not safe for concurrent LoadAsync calls. Internal state (stopwatch, progress counters) assumes a single load in flight. Build a separate instance per concurrent load.

Command timeout uses the Dapper/ADO.NET default (typically 30 seconds). A dedicated CommandTimeout property is planned (see GitHub issue #25).

Properties

BatchCommitSize

Commit the auto-managed transaction every N successfully-loaded rows. Defaults to 0, meaning "commit only once at the end of the load" (the v0.4.0 behavior). Larger values let very long loads survive a mid-flight failure with most of the work persisted.

public int BatchCommitSize { get; set; }

Property Value

int

Remarks

Only takes effect when the loader manages the transaction (i.e. the caller did NOT pass a DbTransaction to the constructor) AND BatchSize is 1. With a caller-supplied transaction the loader never commits, so this knob has no effect; with BatchSize > 1 the per-batch commit boundary is the BatchSize's own batch, which we currently don't subdivide.

Trade-off: a partial-progress failure with BatchCommitSize > 0 leaves earlier committed batches in the destination. You lose all-or- nothing semantics in exchange for resumability and lower undo-log pressure on the database.

Exceptions

ArgumentOutOfRangeException

The assigned value is negative.

BatchSize

Number of records sent per ExecuteAsync call. Defaults to 1 (one round-trip per record). Larger values pass an IEnumerable<TRecord> to Dapper, which amortizes per-call overhead (parameter parsing, command setup) and lets the ADO.NET provider reuse the prepared command across rows.

public int BatchSize { get; set; }

Property Value

int

Remarks

On networked databases (SQL Server, PostgreSQL, MySQL) this is typically the largest single performance lever in the loader. Memory cost is one buffered List<T> of up to BatchSize records at a time.

SkipItemCount and MaximumItemCount are still honored at the per-record granularity — skipped records never enter the batch buffer, and buffering stops accepting new items once the cumulative-plus-buffered count would exceed MaximumItemCount.

Exceptions

ArgumentOutOfRangeException

Thrown when the assigned value is less than 1.

Remarks

Skip only takes effect on the per-record path (BatchSize == 1, the default). A failed batch cannot be safely attributed to a single row without per-item retry, so the load still aborts even in Skip mode when BatchSize > 1. See Skip.

InsertBatchSize

When greater than 1, the loader replaces N per-row INSERTs with a single multi-row INSERT … VALUES (…), (…), … statement. The single biggest single-line perf win available without provider-specific bulk APIs.

public int InsertBatchSize { get; set; }

Property Value

int

Remarks

Contract on CommandText: it MUST end with the literal token VALUES followed by a parenthesized parameter template such as (@FirstName, @LastName, @Age). The loader extracts that template, replicates it once per row in the batch with suffixed parameter names (@FirstName_0, @FirstName_1, …), and rebinds each row's properties to its slot via reflection.

Database parameter limits apply — SQL Server caps at 2,100 params per command, SQLite at 999, MySQL much higher. Choose InsertBatchSize so that InsertBatchSize × columns stays under your provider's limit.

Default 1 preserves the v0.4.0 behavior (one statement per row, or N statements bundled via BatchSize).

Mutually exclusive with BatchSize > 1 — when both are set, InsertBatchSize wins because it generates a single statement per chunk instead of N. Mutually exclusive with IsDryRun = true (IsDryRun short-circuits before SQL generation) and with stored-procedure CommandType.

Exceptions

ArgumentOutOfRangeException

The assigned value is less than 1.

IsDryRun

When true, the loader runs the full pipeline — enumerates the source, evaluates SkipItemCount / MaximumItemCount, increments progress counters, fires the progress-timer callback, and emits all the usual log messages — but skips the actual ExecuteAsync call. The database is not modified.

public bool IsDryRun { get; set; }

Property Value

bool

Remarks

Useful for validating an ETL pipeline (source feed, mapping, batching, throttling) against production without writing anything, or for estimating how long a write would take.

The auto-managed-transaction path still BeginTransaction / Commits — those are connection-level operations and are needed for the open/dispose lifecycle. They have no effect on the database because no writes happen inside the transaction.

Default false preserves the prior behavior. This is the implementation of Wolfgang.Etl.Abstractions.ISupportDryRun.IsDryRun from Wolfgang.Etl.Abstractions 0.15.0+.

ManageConnection

When true, the loader opens the connection before the first command runs and closes it after the load ends. The connection is NOT disposed — it's returned to the pool for reuse, which plays better with connection-pool lifetime in web apps and hosted services.

public bool ManageConnection { get; set; }

Property Value

bool

Remarks

Default false preserves the v0.4.0 behavior: the caller is responsible for opening the connection before calling LoadAsync.

Ignored on the owned-connection ctor path (the (DbProviderFactory, connectionString, …) overload). That path always manages and disposes the connection because it created it.

If the connection is already open when LoadAsync starts, it's left open — the loader only closes connections it itself opened.

MaxErrorCount

Maximum number of row-level failures the loader will tolerate before aborting the load with an InvalidOperationException whose InnerException is the last underlying row failure. Defaults to 0, meaning "unlimited" (every failure is reported via RowFailed and processing continues).
