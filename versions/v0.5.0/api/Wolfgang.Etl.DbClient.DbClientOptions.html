Remarks

Scope today:

  • Applies to DbExtractor<TRecord> — its static constructor triggers the type-map registration that consults these settings.
  • Only takes effect for TRecord types that have at least one ColumnAttribute; types without any [Column] use Dapper's built-in name matcher, which is not routed through this options class.
  • Does not affect DbLoader<TRecord>: loaders emit parameters from a POCO, they don't read result-set columns.

Set the flags once at process startup, before constructing the first extractor for a given TRecord, for the most predictable behavior.

Properties

StrictColumnMapping

Controls how the library reacts when a SELECT result-set column does not map to any property on the destination TRecord type.

public static bool StrictColumnMapping { get; set; }

Property Value

bool

Remarks

  • false (default) — silently ignore the unmapped column, matching Dapper's out-of-the-box behavior. Common with SELECT * and joins that include columns the POCO doesn't need.
  • true — throw an InvalidOperationException naming the offending column and the target type. Useful while developing to catch typos in [Column] attributes the moment they happen.

The flag is read on every column lookup, so flipping it at runtime takes effect for the next query without re-registering type maps.
