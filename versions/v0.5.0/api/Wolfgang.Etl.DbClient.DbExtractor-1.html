Remarks

The caller owns the DbConnection lifetime — the extractor does not open, close, or dispose it. The connection must be open before calling ExtractAsync.

An optional DbTransaction can be provided for isolation level control. The extractor never commits or rolls back the transaction.

Thread safety. A DbExtractor<TRecord> instance is not safe for concurrent ExtractAsync calls. Internal state (stopwatch, total-count snapshot, progress-counter increments) assumes a single extraction in flight. Build a separate instance per concurrent extraction.

Command timeout uses the Dapper/ADO.NET default (typically 30 seconds). A dedicated CommandTimeout property is planned (see GitHub issue #25).

DefaultTotalCountQuery

The default total count implementation. Wraps CommandText in SELECT COUNT(*) FROM (...) AS _count and executes it using the same connection, parameters, and transaction as the extraction query. Assign this to TotalCountQuery to enable the built-in behavior.

public Func<CancellationToken, Task<int>> DefaultTotalCountQuery { get; }

Property Value

Func<CancellationToken, Task<int>>

Remarks

Trailing semicolons are stripped automatically. If the command text contains an ORDER BY clause, some database providers (e.g. SQL Server) may reject it inside a derived table. Use a custom TotalCountQuery in that case.

ManageConnection

When true, the extractor opens the connection before the first command runs and closes it after the enumeration ends. The connection is NOT disposed — it's returned to the pool for reuse, which plays better with connection-pool lifetime in web apps and hosted services.

public bool ManageConnection { get; set; }

Property Value

bool

Remarks

Default false preserves the v0.4.0 behavior: the caller is responsible for opening the connection before calling ExtractAsync.

Ignored on the owned-connection ctor path (the (DbProviderFactory, connectionString, …) overload). That path always manages and disposes the connection because it created it.

If the connection is already open when ExtractAsync starts, it's left open — the extractor only closes connections it itself opened.

PagingClauseTemplate

SQL fragment appended to the command text when both ServerOffset and ServerLimit are set. Bound as Dapper parameters @PageOffset and @PageLimit.

public string PagingClauseTemplate { get; set; }

Property Value

string

Remarks

Defaults to LIMIT @PageLimit OFFSET @PageOffset — the SQLite / PostgreSQL / MySQL syntax.

For SQL Server, set to OFFSET @PageOffset ROWS FETCH NEXT @PageLimit ROWS ONLY (and ensure the base SQL ends with an ORDER BY).

Parameters

Optional override for the parameter set passed to Dapper. Setting this property takes precedence over any IDictionary<string,object> supplied via the constructor — useful when the command is a stored procedure with OUT / INOUT parameters that need to be declared with ParameterDirection values Dapper can't infer from a plain dictionary.

public DynamicParameters? Parameters { get; set; }

Property Value

DynamicParameters

Remarks

Caller-owned: the extractor never clones it. After ExtractAsync completes, read output values via Parameters.Get<T>("@name").

Example: 

var p = new DynamicParameters();
p.Add("@CustomerId", 42);
p.Add("@TotalCount", dbType: DbType.Int32, direction: ParameterDirection.Output);
var extractor = new DbExtractor<Order>(conn, "usp_GetOrdersForCustomer")
{
    CommandType = CommandType.StoredProcedure,
    Parameters = p
};
var orders = await extractor.ExtractAsync().ToListAsync();
var total = p.Get<int>("@TotalCount");

Property Value

ServerOffset

When both ServerOffset and ServerLimit are set, the extractor appends PagingClauseTemplate to the command text before sending it. Default null disables server-side paging (the v0.4.0 behavior — the full result set comes back and SkipItemCount/MaximumItemCount filter client-side).

public long? ServerOffset { get; set; }

Property Value

long?

Remarks

Use server-side paging for very large tables where streaming everything to the client is wasteful. SQL Server requires an ORDER BY in the command text for paging to be deterministic; SQLite, PostgreSQL, and MySQL don't require it but you should still include one — without a stable order, page contents drift.

Remarks

Doesn't mutate any state on the extractor — does not touch DbReport.TotalItemCount, the stopwatch, or any of the progress counters. Safe to call any number of times before, during (different cancellation token), or after an ExtractAsync.

Opens the connection on the owned-connection ctor path before running the query and disposes it after — same lifecycle as a full extraction.
